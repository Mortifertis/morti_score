# Моделирование

Morti Score использует простые статистические футбольные модели, чтобы показать,
как backend может объединять предметные данные, расчёты моделей, кэширование и
схемы ответов API. Модели намеренно сделаны прозрачными и лёгкими: они полезны
для инженерной демонстрации и объяснимости, но не предназначены для ставок или
торговых решений.

## Базовая Poisson-модель

Базовая Poisson-модель рассматривает голы как счётные события. Для каждой
команды она оценивает силу атаки и защиты по историческим завершённым матчам,
а затем объединяет эти показатели со средними домашними и гостевыми
результатами по лиге.

Распределение Poisson часто используют как базовый подход в футболе, потому что
голы являются редкими целочисленными событиями. В этом проекте оно применяется
как интерпретируемая первая модель: рассчитать ожидаемые голы обеих команд,
сгенерировать вероятности счёта и агрегировать их в вероятности победы хозяев,
ничьей и победы гостей.

## Используемые данные

Текущие модели используют только локальные данные проекта:

- команды из таблицы `teams`, загруженные из `data/teams.json`;
- матчи из таблицы `matches`, загруженные из `data/matches.json`;
- только завершённые матчи, где заполнены `home_goals` и `away_goals`;
- даты матчей, идентификаторы команд, счёт и статус матча;
- производные значения, например форма команд, средние по лиге и xG-прокси.

Prediction endpoint не обращается к внешнему поставщику футбольных данных.
Это делает demo воспроизводимым, но ограничивает качество моделей.

## Расчёт ожидаемых голов

Для базовой модели сервис сначала рассчитывает средние значения на уровне лиги:

- среднее число голов хозяев за завершённый матч;
- среднее число голов гостей за завершённый матч.

Затем рассчитываются показатели силы команды:

- домашняя атака: голы команды дома относительно среднего показателя хозяев
  по лиге;
- домашняя защита: пропущенные дома голы относительно среднего показателя
  гостей по лиге;
- гостевая атака: голы команды в гостях относительно среднего показателя
  гостей по лиге;
- гостевая защита: пропущенные в гостях голы относительно среднего показателя
  хозяев по лиге.

Ожидаемые голы оцениваются так:

```text
expected_home_goals = league_home_goals
  * home_team_home_attack
  * away_team_away_defense

expected_away_goals = league_away_goals
  * away_team_away_attack
  * home_team_home_defense
```

Матрица счёта применяет вероятности Poisson для результатов от 0:0 до 5:5 и
возвращает наиболее вероятные варианты счёта, а также нормализованные
вероятности исходов матча.

## Улучшенная Poisson-модель

Улучшенная Poisson-модель сохраняет ту же базовую структуру, но добавляет
текущую форму команд и корректировку через xG-прокси.

Она использует скользящее окно последних матчей каждой команды и применяет вес
по давности, поэтому новые матчи сильнее влияют на оценку, чем старые. Модель
смешивает долгосрочную силу команды с текущей атакующей и оборонительной
формой. Это помогает модели реагировать, когда команда сейчас сильнее или
слабее, чем показывают средние значения за сезон.

## xG-прокси в этом проекте

Настоящие ожидаемые голы требуют данных на уровне ударов: позиция удара, часть
тела, тип передачи, давление соперника, состояние игры и другие контекстные
признаки. Seed-данные этого проекта не содержат таких полей.

Поэтому Morti Score использует xG-прокси, а не настоящий xG. Прокси — это
детерминированная эвристика, построенная из доступных данных матча:

- забитые голы увеличивают атакующий прокси-показатель;
- пропущенные голы добавляют меньшую оборонительную корректировку и поправку на
  состояние игры;
- для домашнего и гостевого контекста используются немного разные множители;
- значение ограничено снизу, чтобы оно не могло схлопнуться к нулю.

Этот прокси полезен, чтобы показать, где xG-подобный сигнал вошёл бы в
архитектуру. Его не следует воспринимать как реальные ожидаемые голы.

## Elo-модель

Elo-модель назначает каждой команде рейтинг и обновляет его после каждого
завершённого матча. Команда получает рейтинговые очки, если выступила лучше
ожиданий, и теряет очки, если выступила хуже ожиданий.

В этом проекте Elo использует:

- базовый рейтинг для новых команд;
- K-factor для управления величиной обновления;
- преимущество домашнего поля в расчёте предматчевого ожидания;
- множитель разницы голов, чтобы крупные победы сильнее меняли рейтинг;
- сортировку матчей по дате и идентификатору перед обновлением рейтингов.

Для прогноза модель переводит разницу рейтингов в ожидаемость победы, а затем
использует её, чтобы сдвинуть средние домашние и гостевые ожидаемые голы по
лиге.

## Почему это не production betting model

Эти модели не являются production betting models, потому что им не хватает
данных, валидации и управления рисками, необходимых для такого сценария.

Среди важных недостающих частей:

- настоящий xG и event-level data об ударах;
- травмы, дисквалификации, составы и информация о ротации;
- рыночные коэффициенты и сравнение с closing line;
- перегруженность календаря дома и в гостях, а также влияние поездок;
- сила команд с поправкой на соперников за несколько сезонов;
- проверки калибровки и backtesting на out-of-sample данных;
- управление банкроллом, риском и неопределённостью;
- мониторинг data drift и сломанных потоков данных.

Текущую реализацию лучше воспринимать как объяснимое analytics demo и пример backend-архитектуры.

## Ограничения подхода с seed-данными

Seed-данные позволяют легко запускать проект, но у них
есть явные ограничения для моделирования:

- набор данных маленький и статичный;
- он может не отражать текущую силу команд;
- в нём нет составов, ударов, владения, коэффициентов и event-level признаков;
- одни и те же данные используются для демонстрации и тестов;
- отсутствует train/test split по времени;
- качество прогнозов нельзя надёжно измерить на такой маленькой выборке.

Это значит, что вывод моделей следует считать иллюстративным, а не точным.

## Как валидировать на реальных данных

Более сильный процесс валидации использовал бы исторические реальные матчи и
строгое разделение по времени:

1. Импортировать несколько сезонов расписаний, результатов и event-level
   данных.
2. Обучать или подбирать параметры модели только на матчах до контрольной даты.
3. Прогнозировать будущие матчи после этой даты без использования информации из
   будущего.
4. Измерять log loss, Brier score и калибровку для вероятностей 1X2.
5. Измерять качество прогноза точного счёта через ranked probability score или
   likelihood.
6. Сравнивать с базовыми подходами, например средним по лиге, Elo-only и
   букмекерскими closing odds там, где это законно доступно.
7. Повторять валидацию по лиге, сезону, разделению дом/гости и уровню силы
   команд.
8. Отслеживать model drift во времени и отправлять alert при ухудшении
   калибровки.

Такой процесс превратил бы текущие demo-модели в измеримый
analytics pipeline.
